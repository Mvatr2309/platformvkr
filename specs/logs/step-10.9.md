# Step 10.9 — Журнал изменений

## Запись #1 — Фикс C1: закрытие GET /api/projects/[id] от анонимного доступа

**Запрос:** Аудит безопасности выявил уязвимость C1 — `GET /api/projects/[id]` не проверял авторизацию и отдавал email/контакты участников и НР. Любой анонимный пользователь мог перебором id выкачивать персональные данные.

**Реализация:**
- Добавлена проверка авторизации в начало GET-хендлера через `requireAuth()` из `lib/api-guard.ts`.
- `requireProjectAccess` намеренно НЕ использован: он запрещает доступ не-участникам, что сломало бы просмотр проекта студентами для подачи заявки (email автора показывается откликающимся by design).
- Остаточный риск (контакты участников команды видны любому авторизованному пользователю) отнесён к M2 — отдельная задача минимизации PII.

**Файлы:**
- `app/api/projects/[id]/route.ts` — импорт `requireAuth, isGuardError`; guard в начале GET.

**Проверка:** `npx tsc --noEmit` — новых ошибок в изменённом файле нет (присутствующие ошибки `favoriteSupervisor`/`.next` предсуществующие, не связаны).

**Коммит:** (не закоммичено — ожидает запроса пользователя)

## Запись #3 — Фикс «Ошибка подачи заявки» при предложении проекта руководителю

**Запрос:** Студентка Панфилова Т.П. не могла предложить свой проект Зверевой А.К. — выходила «Ошибка подачи заявки».

**Причина:** ограничение БД `@@unique([projectId, studentId])` на `Application` разрешает студенту только ОДНУ заявку на проект (любой тип/любой НР). У проекта уже было активное `SUPERVISION_REQUEST` (PENDING) к Атаевой О.М. от 12.05.2026. Предварительная проверка дубликата в коде смотрела только пару (проект + конкретный НР), пропускала запрос к `prisma.create`, а тот падал на нарушении constraint → пустой `catch` отдавал 500 «Ошибка подачи заявки».

**Реализация:**
1. (данные) Удалена устаревшая заявка Атаевой `cmp2fsfkq001xyvwr4j7rvd41` (SUPERVISION_REQUEST/PENDING) на проде — освободило unique-слот.
2. (код) Проверка дубликата в ветке SUPERVISION_REQUEST расширена: ищем любую заявку студента на проект и возвращаем понятный 409 (тому же НР / другому НР с именем / иная активная заявка) вместо падения в 500.
3. (код) В `catch` POST добавлен `console.error` — раньше причина 500 глушилась.

**Файлы:**
- `app/api/applications/route.ts` — расширена проверка дубликата SUPERVISION_REQUEST; логирование в catch.

**Проверка:** `npx tsc --noEmit` — ошибок в файле нет.

**Коммит:** 723e81e (задеплоено)

## Запись #4 — Разрешить предлагать проект нескольким научным руководителям

**Запрос:** Студент должен иметь возможность предложить один проект сразу нескольким НР (раньше из-за `@@unique([projectId, studentId])` — только одному за раз).

**Реализация:**
1. (схема) `Application.@@unique([projectId, studentId])` → `@@unique([projectId, studentId, supervisorId])`. Миграция `20260615120000_allow_multiple_supervision_requests` (DROP старого индекса + CREATE нового).
2. (код, POST /api/applications) ветка SUPERVISION_REQUEST: проверка дубликата теперь по тройке (проект+студент+конкретный НР); добавлен лимит `MAX_SUPERVISION_REQUESTS_PER_PROJECT = 3` активных предложений на проект.
3. (код) STUDENT-ветка: `findUnique({ projectId_studentId })` заменён на `findFirst({ projectId, studentId, type: "STUDENT" })` (составной ключ удалён сменой constraint).
4. Авто-отзыв прочих предложений при выборе одного НР уже был реализован ранее (`student_accept`, updateMany → DECLINED). UI «Мои предложения» уже поддерживал список. Изменений не потребовали.

**Файлы:**
- `prisma/schema.prisma` — изменён `@@unique` на Application.
- `prisma/migrations/20260615120000_allow_multiple_supervision_requests/migration.sql` — новая миграция.
- `app/api/applications/route.ts` — проверка дубликата/лимит SUPERVISION_REQUEST; STUDENT findFirst.

**Проверка:** `npx tsc --noEmit` — ошибок нет.

**Коммит:** 915e024 (задеплоено, миграция применена)

## Запись #6 — Разбор проблем пользователей: добавление тиммейтов, потеря ссылки/участника, кнопка удаления

**Запрос:** разобрать 3 проблемы пользователей и внести исправления.

**Проблема 1 — нельзя добавить тиммейта (роль блокирует):** фронт требовал роль обязательно, бэк — нет; список ролей пуст, если у проекта нет requiredRoles (45% проектов на проде).
→ Роль сделана необязательной в `handleAddManualMember`; плейсхолдер «Роль (необязательно)».

**Проблема 2 — теряется ссылка при создании:** ссылка попадала в проект только через кнопку «Добавить ссылку»; введённая, но не добавленная — терялась на сабмите.
→ В `handleSave` авто-добавление непустой ссылки (`linkUrl`) перед отправкой; валидация http(s) с явной ошибкой вместо потери.

**Проблема 3А — теряется участник при создании:** та же причина для `memberForm`.
→ В `handleSave` авто-добавление заполненного участника; при частично заполненных полях — явная ошибка, данные не теряются.

**Проблема 3Б — пропадает кнопка «Удалить проект»:** by design (нельзя удалить с участниками), но без объяснения.
→ Показывается неактивная кнопка «Удалить» с подсказкой «Сначала удалите участников команды».

**Проблема 3В («взлом»):** не уязвимость — авторизация в members/[memberId] и DELETE /projects/[id] корректна, привилегии не повышаются. Без изменений.

**Файлы:**
- `app/projects/[id]/page.tsx` — роль необязательна; неактивная кнопка удаления с подсказкой.
- `app/projects/new/page.tsx` — авто-коммит ссылки и участника в `handleSave`.

**Проверка:** `npx tsc --noEmit` — ошибок нет.

**Коммит:** 26930ac (задеплоено)

- Список откликов студента (apply-форма) по-прежнему использует `openRoles` — там «свободные слоты» осмысленны, не меняли.

## Запись #9 — Привязка добавленного по почте студента к его аккаунту

**Запрос:** при добавлении участника по почте он помечается «не в системе», даже если студент уже зарегистрирован, и не видит проект.

**Причина:** `manual-members` всегда создавал участника с `inSystem:false` и `manualEmail`, без связи со `studentId` → проект не попадал в «Мои проекты» студента (выборка идёт по `studentId`/`userId`).

**Реализация (`app/api/projects/[id]/manual-members/route.ts`):**
- Перед созданием ищем `StudentProfile` по email пользователя с ролью STUDENT.
- Если найден — участник создаётся со связью `studentId` и `inSystem:true` (manual-поля не заполняются); студенту шлётся уведомление «Вас добавили в проект».
- Если не найден — как раньше, ручной участник.

**Данные (прод):** существующие ручные участники, чья почта совпала с зарегистрированным студентом: 22 привязаны к аккаунтам (studentId, inSystem=true, manual-поля очищены), 3 удалены как дубли (студент уже был настоящим участником проекта). Несвязанных совпадений не осталось (0).

**Файлы:** `app/api/projects/[id]/manual-members/route.ts`.

**Проверка:** `npx tsc --noEmit` — ошибок нет.

**Коммит:** (ожидает деплоя)
